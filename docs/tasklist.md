## Итерационный план разработки

- [x] **Итерация 1 — Базовый Telegram-бот**
  - Минимальный бот на `aiogram` с командой `/start` и простым echo-ответом.
  - Проверка: бот запускается, отвечает на `/start` и текстовые сообщения.

- [x] **Итерация 2 — Приём аудиофайла**
  - Поддержка отправки пользователем аудиофайла, генерация `track_id` и создание папки трека.
  - Проверка: после отправки файла бот отвечает сообщением с `track_id` и путём сохранения.

- [x] **Итерация 3 — Приём произвольной ссылки (кроме `music.yandex.ru`, YouTube)**
  - Поддержка отправки пользователем произвольной ссылки на трек или файл, кроме ссылок на `https://music.yandex.ru/` и YouTube.
  - Проверка: после отправки ссылки бот отвечает сообщением с `track_id` и сведениями о сохранённом источнике.

- [x] **Итерация 4 — Каркас пайплайна без интеграций**
  - Реализация класса пайплайна с шагами (`DOWNLOAD → SEPARATE → TRANSCRIBE → GET_LYRICS → ALIGN → GENERATE_ASS → RENDER_VIDEO`) с заглушками вместо настоящей логики.
  - Проверка: по команде бот запускает последовательность шагов, логирует и сообщает прогресс по каждому шагу.

- [x] **Итерация 5 — Разделение голоса и музыки (`Demucs`)**
  - Подключение `DemucsService` и выполнение шага `SEPARATE` с сохранением двух аудиодорожек.
  - Проверка: для трека после шага появляются файлы с голосом и музыкой, бот сообщает об успешном разделении.

- [x] **Итерация 6 — Транскрибация голосовой дорожки (`speeches.ai`)**
  - Подключение `SpeechesClient` и выполнение шага `TRANSCRIBE` с сохранением результата в JSON.
  - Проверка: для голосовой дорожки создаётся JSON с транскриптом, бот уведомляет о завершении транскрибации.

- [x] **Итерация 7 — Получение текста песни**
  - Реализация шага `GET_LYRICS`: сначала авто-поиск текста через sub-pipeline поставщиков, при неудаче — запрос полного текста у пользователя.
  - Sub-pipeline авто-поиска (`GET_LYRICS`) последовательно пробует следующих поставщиков (каждый включается/отключается через конфигурацию):
    - [x] Поиск через `Lyrica` ([github.com/Wilooper/Lyrica](https://github.com/Wilooper/Lyrica)) — управляется флагом `LYRICS_ENABLE_LYRICA` (по умолчанию `false`)
    - [ ] Поиск через `lyrics-lib` ([npmjs.com/package/lyrics-lib](https://www.npmjs.com/package/lyrics-lib)) — управляется флагом `LYRICS_ENABLE_LYRICSLIB` (по умолчанию `false`)
    - [x] Поиск через `Genius API` (`lyricsgenius`) — управляется флагом `LYRICS_ENABLE_GENIUS` (по умолчанию `true` при наличии `GENIUS_TOKEN`)
  - Если все поставщики вернули `None` — запрос текста у пользователя через FSM (`LyricsStates.waiting_for_lyrics`)
  - [x] Добавить в `app/config.py` параметры: `LYRICS_ENABLE_LYRICA`, `LYRICS_ENABLE_LYRICSLIB`, `LYRICS_ENABLE_GENIUS`
  - [x] Реализовать `LyricsService` с поддержкой `Genius API` (`lyricsgenius`)
  - [x] Реализовать `LyricsNotFoundError` и обработку в `KaraokePipeline._step_get_lyrics()`
  - [x] Реализовать FSM-хендлер `handle_lyrics_input` для ручного ввода текста
  - [x] Реализовать поставщика `Lyrica` в `LyricsService` (когда `LYRICS_ENABLE_LYRICA=true`)
  - [ ] Реализовать поставщика `lyrics-lib` в `LyricsService` (когда `LYRICS_ENABLE_LYRICSLIB=true`)
  - Проверка: бот либо сам находит текст (через одного из включённых поставщиков), либо корректно запрашивает и сохраняет текст у пользователя, сообщает результат.

- [x] **Итерация 8 — Выравнивание текста по таймкодами**
  - Шаг `ALIGN`: сопоставление текста с таймкодами слов на основе транскрипта (с использованием LLM при необходимости). Детальный план сервиса в @docs/align.md
  - Реализован `AlignmentService` с двумя стратегиями: `LrcDirectStrategy` (при наличии LRC-таймкодов в тексте) и `SequenceAlignmentStrategy` (алгоритм Нидлмана–Вунша).
  - Интегрирован в `KaraokePipeline._step_align()`.
  - Проверка: создаётся файл с выравненным текстом и таймкодами, бот сообщает об успешном выравнивании.
  

- [x] **Итерация 9 — Генерация субтитров `.ass`**
  - Шаг `GENERATE_ASS`: генерация файла субтитров `.ass` на основе выровненного текста и таймкодов.
  - Реализован `AssGenerator` (`app/ass_generator.py`) с поддержкой: заголовка трека, per-segment TextLine, per-word Highlight с тегами `{\rHighlight}…{\rDefault}`.
  - Интегрирован в `KaraokePipeline._step_generate_ass()`.
  - Добавлен параметр конфигурации `ASS_FONT_SIZE` (по умолчанию `60`).
  - Проверка: в папке трека появляется корректный `.ass`-файл, бот сообщает об успешной генерации.

- [x] **Итерация 10 — Рендеринг караоке-видео**
  - Шаг `RENDER_VIDEO`: рендеринг простого караоке-видео через `ffmpeg` с использованием `.ass` и аудиодорожки.
  - Реализован `VideoRenderer` (`app/video_renderer.py`) с синтетическим цветным фоном (`lavfi color`) и сжиганием субтитров через фильтр `ass`.
  - Добавлены параметры конфигурации `VIDEO_WIDTH`, `VIDEO_HEIGHT`, `VIDEO_BACKGROUND_COLOR`, `VIDEO_FFMPEG_PRESET`, `VIDEO_FFMPEG_CRF`.
  - Интегрирован в `KaraokePipeline._step_render_video()`.
  - Бот отправляет итоговый MP4-файл пользователю через `answer_video` после завершения пайплайна.
  - Проверка: бот присылает пользователю итоговый MP4-файл с базовыми караоке-субтитрами.

- [x] **Итерация 11 — Продолжение после ошибок**

- [x] **Итерация 12 — Корректировка таймингов в шаге ALIGN**
  - Добавить в шаг `ALIGN` метод, который запускается после выполнения сопоставления строк и слов с таймингами.
  - Метод должен проверить длительность первых слов в строках, и если длительность первого слова более "Max_word_time", то выполняется:
    1) Сохраняем "start_time_gap" ="start_time", "start_time" присваивается "end_time"-"normal_word_time"
    2) Перед первым словом в строке добавляется "(проигрыш)", у которого "start_time"="start_time_gap"
    3) В разделе Segments также вносим корректировку в соответствующую строку добавив перед первым словом "(проигрыш)"
  - Пример:
    Words:  [{
          "word": "myself",
          "start_time": 53.18,
          "end_time": 54.96
        },
    #начало добавленного блока
        {
          "word": "(проигрыш)",
          "start_time": 54.96,
          "end_time": 59.4
        },
    #конец добавленного блока
        {
          "word": "Tell",
          "start_time": 59.4,
          "end_time": 60.6
        },
    ],
    segments:[
        {
          "text": "And, in the bad times, I fear myself",
          "start_time": 48.58,
          "end_time": 54.96
        },
        {
          "text": "(проигрыш) Tell me something, boy",
          "start_time": 54.96,
          "end_time": 68.14
        },
    ]
  - Добавить параметры в .env:
    - "MAX_WORD_TIME" = 5 секунд  # время максимально допустимой длительности первого слова в строке
    - "NORMAL_WORD_TIME" # нормальное время длительности первого слова в строке
  - Проверка: при наличии слов с длительностью более MAX_WORD_TIME в начале строк, происходит корректировка таймингов с добавлением "(проигрыш)" участков.

- [x] **Итерация 17 — Корректировка транскрипции с использованием LLM (CORRECT_TRANSCRIPT)**
  - Добавить новый шаг CORRECT_TRANSCRIPT в пайплайн, который выполняется после TRANSCRIBE и перед ALIGN
  - Шаг использует LLM для корректировки транскрипции на основе загруженного текста песни
  - Цель: улучшить качество выравнивания текста путём предварительной корректировки распознанного текста
  - **План реализации:**
    1. Добавить параметры конфигурации для LLM (провайдер, API ключ, модель)
    2. Реализовать сервис LLM-корректировки транскрипции (CorrectTranscriptService)
    3. Интегрировать шаг CORRECT_TRANSCRIPT в KaraokePipeline
    4. Шаг принимает на вход:
       - Готовую транскрипцию в json(из шага TRANSCRIBE)
       - Загруженный текст песни (из шага GET_LYRICS)
    5. LLM анализирует транскрипцию и текст песни, возвращает скорректированную транскрипцию в json того же формата, что и исходная транскрипция
    6. Скорректированный текст сохраняется и передаётся на шаг ALIGN
  - **Проверка:** после выполнения шага CORRECT_TRANSCRIPT создаётся скорректированная транскрипция, которая затем используется в шаге ALIGN для более точного выравнивания.
