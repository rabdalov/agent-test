# Итерация 48: Оценка возможности перехода на Conda вместо UV

## Цель
Провести анализ целесообразности перехода с `uv` на `conda`/`mamba` для управления зависимостями проекта на основе **анализа документации** (без реального тестирования).

## Методология

**Только документальный анализ:**
- Изучение документации UV, Conda, Mamba
- Анализ best practices для ML-проектов
- Сравнение по публичным бенчмаркам и отчётам
- Изучение совместимости зависимостей на уровне спецификаций

## Исходные данные

### Текущий стек (UV)
- `pyproject.toml` + `uv.lock`
- `uv sync --frozen --no-dev` для сборки
- `python:3.12-slim` в Docker

### ML-зависимости проекта
- `demucs>=4.0.0` — требует torch
- `librosa>=0.11.0` — требует numpy, scipy
- `msaf>=0.1.7` — требует scikit-learn, C-extensions
- `torch` (implicit через demucs)

### Будущая зависимость (критично!)
- **`mmcauliffe/montreal-forced-aligner`** (MFA) — запланирована к добавлению
- **Проблема:** MFA **не запускается под UV** — требует conda-окружения
- MFA имеет сложные нативные зависимости (kaldi, openfst) с conda-only пакетами
- Это меняет оценку UV vs Conda!

## Анализ по критериям

### 1. Скорость установки (по документации)

| Инструмент | Скорость | Источник |
|------------|----------|----------|
| UV | ~10-50x быстрее pip | astral.sh блог |
| Conda | Медленнее pip | conda docs |
| Mamba | ~3-5x быстрее conda | mamba docs |
| UV + Conda | UV быстрее для wheels | сравнительные тесты |

**Вывод:** UV значительно быстрее для Python-only пакетов.

### 2. Поддержка ML-экосистемы

| Критерий | UV | Conda |
|----------|-----|-------|
| PyTorch с CUDA/ROCm | Нужен `--extra-index-url` | Нативная поддержка `pytorch-cuda` или `pytorch-rocm` |
| PyTorch на AMD GPU | ROCm через pip | ROCm через `pytorch-rocm` канал |
| Бинарные wheels | Да | Да (conda-каналы) |
| Scientific libs | Через wheels | Оптимизированные MKL/OpenBLAS |
| demucs | Устанавливается | Устанавливается |

**AMD GPU специфика:**
- AMD Rembrandt 680M поддерживает **ROCm 5.7+** (частичная поддержка iGPU)
- PyTorch для ROCm доступен и через pip, и через conda
- MFA на ROCm — не тестировано, вероятно требует CPU fallback

**Документация PyTorch:** рекомендует Conda для GPU-версий из-за автоматического управления CUDA/ROCm toolkit.

### 3. Размер Docker-образа

| Подход | Размер | Источник |
|--------|--------|----------|
| `python:3.12-slim` + UV | ~200-300 MB база | Docker Hub |
| `continuumio/miniconda3` | ~400-500 MB база | Docker Hub |
| `debian:13-slim` + conda | ~300-400 MB | расчётная |

**Вывод:** UV даёт меньший размер образа.

### 4. GPU-поддержка (важно!)

**Важно:** У пользователя **AMD GPU** (Rembrandt Radeon 680M), не NVIDIA!

#### PyTorch на AMD GPU (ROCm)

##### UV подход:
```dockerfile
# ROCm версия для AMD GPU
RUN uv pip install torch --extra-index-url https://download.pytorch.org/whl/rocm6.0
```
- Ручное управление ROCm версией
- ROCm runtime должен быть на хосте

##### Conda подход:
```dockerfile
RUN conda install pytorch pytorch-rocm -c pytorch
```
- Автоматическая установка ROCm-зависимостей
- Проще управление версиями

**Документация AMD:** ROCm в Conda стабильнее чем pip для сложных зависимостей.

#### Ограничения AMD Rembrandt 680M:
- iGPU имеет **частичную поддержку ROCm 5.7+**
- Не все операции PyTorch оптимизированы для iGPU
- Для demucs: ROCm может не дать прироста или работать хуже CPU
- MFA: не тестировано на ROCm, вероятен fallback на CPU

#### Сравнение GPU поддержки

| Функция | NVIDIA CUDA | AMD ROCm | CPU |
|---------|-------------|----------|-----|
| PyTorch | ✅ Полная | ⚠️ Частичная (iGPU) | ✅ Полная |
| demucs | ✅ Хорошая | ⚠️ Неизвестно | ✅ Хорошая |
| MFA | ✅ Поддержка | ❓ Неизвестно | ✅ Поддержка |
| Контейнеры | ✅ nvidia-docker | ⚠️ Сложнее | ✅ Любые |

**Вывод для AMD iGPU:** GPU-ускорение **не критично** и может быть нестабильным. CPU-only — надёжный выбор.

### 5. Воспроизводимость

| Критерий | UV | Conda |
|----------|-----|-------|
| Lock-файл | `uv.lock` — детерминированный | `conda-lock.yml` или `environment.yml` |
| Кросс-платформа | Да | Ограниченная |
| Скорость resolve | Мгновенно | Медленно |

**Вывод:** UV лучше для воспроизводимости.

## Специфика проекта

### Montreal Forced Aligner (MFA) — критический фактор

**Проблема:**
- MFA распространяется **только через conda** ([conda-forge](https://anaconda.org/conda-forge/montreal-forced-aligner))
- Имеет зависимости: kaldi, openfst, pynini — сложные C++ библиотеки
- **Не работает с чистым pip/UV** — требует conda-окружения

**Варианты интеграции MFA:**
1. **Гибридный подход** (рекомендуется):
   - Основной проект на UV
   - MFA в отдельном conda-окружении
   - Вызов MFA через subprocess

2. **Полный переход на Conda:**
   - Весь проект в conda
   - MFA + все остальные зависимости в одном окружении

3. **Отказ от MFA:**
   - Оставить текущий alignment (без MFA)

### Нужна ли GPU?

**Анализ зависимостей:**
- `demucs` — **работает на CPU** (default), GPU опционально
- `torch` в demucs — используется для inference
- `librosa`, `msaf` — только CPU
- `MFA` — может использовать GPU для акустической модели

**Производительность:**
- Demucs на CPU: ~1-2x realtime (приемлемо для бота)
- Demucs на GPU: ~10-20x realtime
- MFA alignment: быстрее на GPU для длинных файлов

**Вывод:** GPU **не критичен** для текущего функционала, но ускорит MFA в будущем.

### Системные зависимости

**UV + Debian:**
```dockerfile
RUN apt-get install ffmpeg libsndfile1
```

**Conda:**
```dockerfile
RUN conda install ffmpeg librosa  # ffmpeg в conda-forge
```

Оба варианта работают.

## Результаты анализа

### Сценарий A: Гибридный подход с MFA (РЕКОМЕНДУЕТСЯ)

**Обоснование:**
1. MFA требует conda — это неизбежно
2. UV быстрее и компактнее для остальных зависимостей
3. Разделение ответственности: UV — Python, Conda — MFA

**Архитектура:**
```
┌─────────────────┐     ┌──────────────────┐
│   Main App      │────▶│  MFA Service     │
│   (UV + Python) │     │  (Conda env)     │
└─────────────────┘     └──────────────────┘
        │                        │
        ▼                        ▼
   pyproject.toml          environment.yml
```

**Реализация:**
- Основной проект остаётся на UV
- MFA устанавливается в отдельный conda-env
- Взаимодействие через CLI (`mfa align ...`)
- Docker: multi-stage с conda-слоем для MFA

### Сценарий B: Полный переход на Conda

**Когда имеет смысл:**
- Если MFA используется интенсивно (каждый запрос)
- Если важна единая среда
- Если планируется GPU для MFA

**Недостатки:**
- Медленнее resolve зависимостей
- Больше размер образа
- Хуже lock-файлы

### Сценарий C: Отказ от MFA

**Если:**
- Текущий alignment достаточен
- Не хочется усложнять инфраструктуру

## Итоговая рекомендация

> **Гибридный подход:** UV для основного проекта + Conda для MFA в отдельном сервисе/окружении.

Это позволяет:
1. Сохранить скорость и компактность UV
2. Использовать MFA без костылей
3. Иметь чёткое разделение компонентов

**Действия:**
1. Оставить текущую UV-структуру
2. Добавить `mfa/` директорию с `environment.yml`
3. Создать обёртку `MfaService` для вызова MFA
4. Обновить Docker для поддержки двух окружений
