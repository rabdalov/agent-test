# Анализ: UV vs Conda для ML-проекта

**Дата:** 2026-03-19
**Итерация:** 48
**Тип:** Документальный анализ (без реального тестирования)

---

## Архитектурное решение

> **Статус:** ✅ Принято
> **Решение:** Использовать **только UV**, отказаться от MFA
> **Обоснование:** Путь максимальной стабильности

### Обоснование решения

Несмотря на то что Montreal Forced Aligner (MFA) потенциально мог бы улучшить качество alignment, интеграция требует:
- Введения conda-окружения в дополнение к UV
- Усложнения инфраструктуры (multi-stage Docker, два менеджера зависимостей)
- Потери воспроизводимости и скорости сборки

**Приоритет:** Простота и стабильность > максимальное качество alignment.

Текущий [`AlignmentService`](app/alignment_service.py) с LRC-импортом и sequence alignment достаточен для задач проекта.

## 1. Исходные данные проекта

### Текущий стек (UV)
- `pyproject.toml` + `uv.lock`
- `uv sync --frozen --no-dev` для сборки
- Базовый образ: `python:3.12-slim`

### ML-зависимости
| Пакет | Версия | Особенности |
|-------|--------|-------------|
| `demucs` | >=4.0.0 | Требует torch, работает на CPU |
| `librosa` | >=0.11.0 | numpy, scipy |
| `msaf` | >=0.1.7 | scikit-learn, C-extensions |
| `torch` | implicit | через demucs |

### Критическая будущая зависимость
**`mmcauliffe/montreal-forced-aligner` (MFA)** — запланирована к добавлению.

**Проблема:** MFA **не работает с чистым pip/UV** — требует conda-окружения.
- Распространяется только через [conda-forge](https://anaconda.org/conda-forge/montreal-forced-aligner)
- Зависимости: kaldi, openfst, pynini — сложные C++ библиотеки
- Conda-only пакеты без аналогов в PyPI

## 2. Сравнение по критериям

### 2.1 Скорость установки

| Инструмент | Скорость | Источник |
|------------|----------|----------|
| **UV** | ~10-50x быстрее pip | astral.sh блог |
| Conda | Медленнее pip | conda docs |
| Mamba | ~3-5x быстрее conda | mamba docs |

**Вывод:** UV значительно быстрее для Python-only пакетов.

### 2.2 Поддержка ML-экосистемы

| Критерий | UV | Conda |
|----------|-----|-------|
| PyTorch с CUDA | `--extra-index-url` | Нативная `pytorch-cuda` |
| Бинарные wheels | Да | Да (conda-каналы) |
| Scientific libs | Через wheels | Оптимизированные MKL/OpenBLAS |
| demucs | ✅ Устанавливается | ✅ Устанавливается |
| MFA | ❌ Не поддерживается | ✅ Только conda-forge |

**Примечание:** Документация PyTorch рекомендует Conda для GPU-версий.

### 2.3 Размер Docker-образа

| Подход | Размер базового образа |
|--------|------------------------|
| `python:3.12-slim` + UV | ~200-300 MB |
| `continuumio/miniconda3` | ~400-500 MB |
| `debian:13-slim` + conda | ~300-400 MB |

**Вывод:** UV даёт меньший размер образа.

### 2.4 GPU-поддержка

**UV подход:**
```dockerfile
RUN uv pip install torch --extra-index-url https://download.pytorch.org/whl/cu121
```
- Ручное управление CUDA версией
- Нужен отдельный CUDA toolkit или `nvidia/cuda` образ

**Conda подход:**
```dockerfile
RUN conda install pytorch pytorch-cuda=12.1 -c pytorch -c nvidia
```
- Автоматическая установка CUDA toolkit
- Согласованные версии CUDA/пакетов

**Вывод:** Conda упрощает управление CUDA (по данным NVIDIA).

### 2.5 Воспроизводимость

| Критерий | UV | Conda |
|----------|-----|-------|
| Lock-файл | `uv.lock` — детерминированный | `conda-lock.yml` / `environment.yml` |
| Кросс-платформа | ✅ Да | ⚠️ Ограниченная |
| Скорость resolve | Мгновенно | Медленно |

**Вывод:** UV лучше для воспроизводимости.

## 3. Анализ проблемы MFA

### Почему MFA требует Conda?

MFA (Montreal Forced Aligner) — инструмент для forced alignment речи:
- **kaldi** — speech recognition toolkit (C++)
- **openfst** — библиотека конечных автоматов (C++)
- **pynini** — Python-обёртка для OpenFst

Эти зависимости:
- Не имеют wheels в PyPI
- Требуют сложной сборки из исходников
- Доступны как готовые conda-пакеты в conda-forge

### Варианты интеграции MFA

| Вариант | Описание | Плюсы | Минусы |
|---------|----------|-------|--------|
| **A. Гибридный** (рекомендуется) | UV для основного проекта, Conda для MFA | Скорость UV, работа MFA | Два окружения |
| **B. Полный Conda** | Весь проект в conda | Единая среда | Медленнее, больше образ |
| **C. Без MFA** | Оставить текущий alignment | Простота | Ограниченная функциональность |

## 5. Итоговая рекомендация

### Гибридный подход (Сценарий A)

**Архитектура:**
```
┌─────────────────┐     ┌──────────────────┐
│   Main App      │────▶│  MFA Service     │
│   (UV + Python) │     │  (Conda env)     │
└─────────────────┘     └──────────────────┘
         │                        │
         ▼                        ▼
    pyproject.toml          environment.yml
```

**Обоснование:**
1. MFA требует conda — это неизбежно
2. UV быстрее и компактнее для остальных зависимостей
3. Чёткое разделение ответственности: UV — Python, Conda — MFA

### Реализация

**Основной проект (UV):**
- Оставить текущую структуру `pyproject.toml`
- Все зависимости кроме MFA через `uv add`

**MFA (Conda):**
```yaml
# mfa/environment.yml
name: mfa-env
channels:
  - conda-forge
  - defaults
dependencies:
  - python=3.11
  - montreal-forced-aligner
  - pip
```

**Взаимодействие:**
```python
# app/mfa_service.py
import subprocess

class MfaService:
    def align(self, audio_path: str, text_path: str, output_path: str) -> None:
        subprocess.run([
            "conda", "run", "-n", "mfa-env",
            "mfa", "align", audio_path, text_path, output_path
        ], check=True)
```

**Docker (multi-stage):**
```dockerfile
# Stage 1: MFA environment
FROM continuumio/miniconda3 AS mfa-base
RUN conda create -n mfa-env -c conda-forge montreal-forced-aligner

# Stage 2: Main application
FROM python:3.12-slim
COPY --from=mfa-base /opt/conda /opt/conda
# ... UV setup
```

## 7. Заключение

### Итоговые принципы

1. **Только UV** — единственный менеджер зависимостей
   Никаких гибридов, никакой conda

2. **Без MFA** — текущий [`AlignmentService`](app/alignment_service.py) достаточен
   LRC-импорт + sequence alignment покрывают потребности проекта

3. **Максимальная стабильность** — простота инфраструктуры приоритетнее потенциального улучшения alignment

### Почему не гибридный подход?

| Фактор | Влияние |
|--------|---------|
| Сложность | Два менеджера зависимостей, multi-stage Docker |
| Поддержка | Сложнее отлаживать, больше точек отказа |
| Выгода | Незначительное улучшение alignment |
| Воспроизводимость | Потеря детерминированности |

### Результат

- `pyproject.toml` + `uv.lock` — единственный источник зависимостей
- `python:3.12-slim` — базовый образ Docker
- Никаких conda-only пакетов в проекте
