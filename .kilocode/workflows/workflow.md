# workflow.md

Итерационный workflow по тасклисту

## Шаги

- **Работать строго по плану**: двигаться итерациями сверху вниз по `docs/tasklist.md`, не перескакивать и не расширять объём без явного запроса пользователя, соотносить решения с `docs/vision.md`.
- **Перед каждой итерацией**: выбирать следующую незавершённую итерацию, кратко формулировать план шагов и предлагать пользователю предполагаемое решение с указанием файлов и примерных фрагментов кода, которые будут изменены/добавлены.
- **Согласование перед реализацией**: дожидаться явного согласия пользователя на предложенный план (включая выбранные файлы и отрезки кода) и только после этого переходить к изменению кода. ВСЕГДА ждать явного "ок" от пользователя.
- **Реализация по согласованному плану**: реализовывать изменения строго по согласованным шагам, придерживаясь принципа KISS и архитектуры из `docs/vision.md`, избегать побочных рефакторингов.
- **После реализации**: кратко описывать сделанные изменения (с ссылками на ключевые файлы и участки кода), задавать пользователю прямой вопрос о приёмке результата и ждать явного подтверждения или комментариев.
- **Обновление прогресса в тасклисте**: после приёмки пользователем обновлять статус текущей итерации в `docs/tasklist.md` (чекбоксы и таблицу прогресса) так, чтобы состояние файла соответствовало фактическому прогрессу.
- **Переход к следующей итерации**: не начинать следующую итерацию, пока не согласовано завершение текущей; перед переходом явно проговаривать, к какой именно итерации переходим дальше.
- **Коммиты в репозиторий**: по завершении каждой итерации выполнять отдельный осмысленный коммит, отражающий суть сделанных изменений, не объединяя в один коммит работы из разных итераций.
