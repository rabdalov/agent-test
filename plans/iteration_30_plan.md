## Проблема

Текущая архитектура имеет следующие недостатки:

- Логика загрузки трека разбросана по 5 разным методам хендлера (`_handle_media_file`, `_handle_yandex_music_url`, `_handle_youtube_url`, `_handle_local_track`, HTTP-загрузка в `handle_text`) — дублирование кода, сложность поддержки.
- Шаги `ASK_LANGUAGE` и `SEND_VIDEO` реализованы вне pipeline в хендлерах, что нарушает принцип линейности и прозрачности.
- Pipeline не является самодостаточным: хендлер управляет FSM-состоянием, уведомлениями и логикой возобновления — это усложняет `handlers_karaoke.py` (>2300 строк).
- Нет единой точки входа в pipeline для всех типов источников.

---

## Цель

Сделать pipeline простым, линейным и самодостаточным. Хендлер должен только принять запрос, создать `PipelineState` и передать управление pipeline.

---

## Ключевые требования к архитектуре

### Изоляция состояния по пользователям

Состояние pipeline уникально для каждого пользователя — несколько пользователей должны одновременно иметь возможность работать каждый со своим треком, не мешая друг другу. `PipelineState` всегда содержит `user_id`, и все операции поиска/возобновления фильтруются по `user_id`.

### Запуск с любого шага

Pipeline должен уметь стартовать с любого шага командой `/step_<name>` (например, `/step_render`, `/step_align`). При этом pipeline проверяет наличие необходимых артефактов для выбранного шага и сообщает об ошибке, если они отсутствуют.

### Работа с последним треком пользователя

Pipeline всегда работает с последним треком, инициированным данным пользователем. Команды `/continue`, `/step_*` автоматически находят последний трек пользователя по `user_id` и `mtime` файла `state.json`.

### Активация трека через /search

Выбор трека в результате команды `/search` активирует найденный трек для pipeline данного пользователя — после выбора все последующие команды (`/continue`, `/step_*`) работают с этим треком.

## Новые шаги pipeline

### `DOWNLOAD` — унифицированный шаг загрузки

Поддерживает все источники:

- Telegram-файл (аудио/видео)
- Локальный файл (из поиска или файловой системы)
- Произвольный HTTP(S) URL
- Яндекс Музыка (`music.yandex.ru`)
- YouTube

Тип источника определяется по полю `source_type` в `PipelineState`.

### `ASK_LANGUAGE` — шаг запроса языка у пользователя

- Pipeline приостанавливается, сохраняет состояние `WAITING_FOR_INPUT`
- Хендлер отправляет inline-клавиатуру выбора языка
- После выбора пользователем — pipeline возобновляется со следующего шага
- Шаг пропускается, если `lang` уже задан в `PipelineState`

### `SEND_VIDEO` — финальный шаг отправки результата пользователю

- Отправка MP4-файла или ссылки на скачивание
- Управляется конфигурацией `SEND_VIDEO_TO_USER`
- Дублирование результата администратору при `user_id != ADMIN_ID`

---

## Дополнительные шаги для повышения надёжности (предложения)

### `VALIDATE` — валидация входных данных и артефактов перед запуском

- Проверка существования исходного файла
- Проверка длительности (не менее 60 секунд)
- Нормализация `track_stem` и создание директории трека

### `CLEANUP` (опционально) — очистка временных файлов после завершения

- Удаление промежуточных файлов (vocal, instrumental) по конфигурации
- Освобождение дискового пространства

---

## Итоговая последовательность шагов pipeline

```
DOWNLOAD → VALIDATE → ASK_LANGUAGE → GET_LYRICS → SEPARATE → TRANSCRIBE
→ CORRECT_TRANSCRIPT → ALIGN → GENERATE_ASS → RENDER_VIDEO → SEND_VIDEO
```

## Изменения в `handlers_karaoke.py`

Все обработчики входящих сообщений (файл, URL, Яндекс, YouTube) сводятся к единому методу `_start_pipeline(message, source_type, source)`:

1. Создать `PipelineState` с `source_type`, `source` и `user_id`
2. Сохранить `state.json`
3. Запустить `KaraokePipeline.run()`

Дополнительные изменения:

- Обработчик выбора языка (`handle_lang_choice`) только обновляет `lang` в `state.json` и вызывает `pipeline.resume()`
- Хендлер перестаёт управлять уведомлениями — это задача pipeline через callback
- Команды `/continue` и `/step_*` ищут последний трек пользователя по `user_id` (фильтрация по `user_id` в `state.json`)
- Выбор трека через `/search` сохраняет `track_id` как "активный" для данного пользователя, чтобы последующие команды `/step_*` работали с этим треком

## Изменения в `pipeline.py`

- Добавить метод `resume(from_step)` для возобновления после `WAITING_FOR_INPUT`
- Шаг `DOWNLOAD` реализует стратегию загрузки на основе `source_type`
- Каждый шаг сам обновляет `PipelineState` и вызывает progress-callback
- При запуске с конкретного шага (`/step_*`) — проверять наличие артефактов предыдущих шагов через `_validate_artifacts_for_step()`

## Проверка

- Все типы источников (файл, URL, Яндекс, YouTube, локальный) проходят через единый `DOWNLOAD`-шаг
- Pipeline линеен: нет ветвлений в хендлере, только в шагах pipeline
- `handlers_karaoke.py` сокращается до ~500 строк (только маршрутизация и FSM)
- Возобновление pipeline после `ASK_LANGUAGE` работает корректно
- Все существующие сценарии (поиск, `/continue`, `/step_*`) продолжают работать
- Два разных пользователя могут одновременно обрабатывать разные треки без взаимного влияния
- Команды `/step_*` применяются к последнему треку данного пользователя
- После выбора трека через `/search` команды `/step_*` работают с выбранным треком
