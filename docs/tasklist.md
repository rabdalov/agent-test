- [x] **Итерация 12 — Корректировка таймингов в шаге ALIGN**
  - Добавить в шаг `ALIGN` метод, который запускается после выполнения сопоставления строк и слов с таймингами.
  - Метод должен проверить длительность первых слов в строках, и если длительность первого слова более "Max_word_time", то выполняется:
    1) Сохраняем "start_time_gap" ="start_time", "start_time" присваивается "end_time"-"normal_word_time"
    2) Перед первым словом в строке добавляется "(проигрыш)", у которого "start_time"="start_time_gap"
    3) В разделе Segments также вносим корректировку в соответствующую строку добавив перед первым словом "(проигрыш)"
  - Пример:
    Words:  [{
          "word": "myself",
          "start_time": 53.18,
          "end_time": 54.96
        },
    #начало добавленного блока
        {
          "word": "(проигрыш)",
          "start_time": 54.96,
          "end_time": 59.4
        },
    #конец добавленного блока
        {
          "word": "Tell",
          "start_time": 59.4,
          "end_time": 60.6
        },
    ],
    segments:[
        {
          "text": "And, in the bad times, I fear myself",
          "start_time": 48.58,
          "end_time": 54.96
        },
        {
          "text": "(проигрыш) Tell me something, boy",
          "start_time": 54.96,
          "end_time": 68.14
        },
    ]
  - Добавить параметры в .env:
    - "MAX_WORD_TIME" = 5 секунд  # время максимально допустимой длительности первого слова в строке
    - "NORMAL_WORD_TIME" # нормальное время длительности первого слова в строке
  - Проверка: при наличии слов с длительностью более MAX_WORD_TIME в начале строк, происходит корректировка таймингов с добавлением "(проигрыш)" участков.

- [x] **Итерация 17 — Корректировка транскрипции с использованием LLM (CORRECT_TRANSCRIPT)**
  - Добавить новый шаг CORRECT_TRANSCRIPT в пайплайн, который выполняется после TRANSCRIBE и перед ALIGN
  - Шаг использует LLM для корректировки транскрипции на основе загруженного текста песни
  - Цель: улучшить качество выравнивания текста путём предварительной корректировки распознанного текста
  - **План реализации:**
    1. Добавить параметры конфигурации для LLM (провайдер, API ключ, модель)
    2. Реализовать сервис LLM-корректировки транскрипции (CorrectTranscriptService)
    3. Интегрировать шаг CORRECT_TRANSCRIPT в KaraokePipeline
    4. Шаг принимает на вход:
       - Готовую транскрипцию в json(из шага TRANSCRIBE)
       - Загруженный текст песни (из шага GET_LYRICS)
    5. LLM анализирует транскрипцию и текст песни, возвращает скорректированную транскрипцию в json того же формата, что и исходная транскрипция
    6. Скорректированный текст сохраняется и передаётся на шаг ALIGN
  - **Проверка:** после выполнения шага CORRECT_TRANSCRIPT создаётся скорректированная транскрипция, которая затем используется в шаге ALIGN для более точного выравнивания.

- [x] **Итерация 30 — Стандартизация pipeline: рефакторинг шагов и унификация загрузки**
  - **Проблема:** текущая архитектура имеет следующие недостатки:
    - Логика загрузки трека разбросана по 5 разным методам хендлера (`_handle_media_file`, `_handle_yandex_music_url`, `_handle_youtube_url`, `_handle_local_track`, HTTP-загрузка в `handle_text`) — дублирование кода, сложность поддержки.
    - Шаги `ASK_LANGUAGE` и `SEND_VIDEO` реализованы вне pipeline в хендлерах, что нарушает принцип линейности и прозрачности.
    - Pipeline не является самодостаточным: хендлер управляет FSM-состоянием, уведомлениями и логикой возобновления — это усложняет `handlers_karaoke.py` (>2300 строк).
    - Нет единой точки входа в pipeline для всех типов источников.
  - **Цель:** сделать pipeline простым, линейным и самодостаточным. Хендлер должен только принять запрос, создать `PipelineState` и передать управление pipeline.
  - **Ключевые требования к архитектуре:**
    - **Изоляция состояния по пользователям:** состояние pipeline уникально для каждого пользователя — несколько пользователей должны одновременно иметь возможность работать каждый со своим треком, не мешая друг другу. `PipelineState` всегда содержит `user_id`, и все операции поиска/возобновления фильтруются по `user_id`.
    - **Запуск с любого шага:** pipeline должен уметь стартовать с любого шага командой `/step_<name>` (например, `/step_render`, `/step_align`). При этом pipeline проверяет наличие необходимых артефактов для выбранного шага и сообщает об ошибке, если они отсутствуют.
    - **Работа с последним треком пользователя:** pipeline всегда работает с последним треком, инициированным данным пользователем. Команды `/continue`, `/step_*` автоматически находят последний трек пользователя по `user_id` и `mtime` файла `state.json`.
    - **Активация трека через /search:** выбор трека в результате команды `/search` активирует найденный трек для pipeline данного пользователя — после выбора все последующие команды (`/continue`, `/step_*`) работают с этим треком.
  - **Новые шаги pipeline:**
    - `DOWNLOAD` — унифицированный шаг загрузки, поддерживающий все источники:
      - Telegram-файл (аудио/видео)
      - Локальный файл (из поиска или файловой системы)
      - Произвольный HTTP(S) URL
      - Яндекс Музыка (`music.yandex.ru`)
      - YouTube
      - Тип источника определяется по полю `source_type` в `PipelineState`
    - `ASK_LANGUAGE` — шаг запроса языка у пользователя:

      - Хендлер отправляет inline-клавиатуру выбора языка

  - **Дополнительные шаги для повышения надёжности (предложения):**
    - `VALIDATE` — валидация входных данных и артефактов перед запуском:

      - Нормализация `track_stem` и создание директории трека
    - `CLEANUP` (опционально) — очистка временных файлов после завершения:

    - Нет порогов по длительности и позиционных критериев.
    - `msaf` требует monkey-patch для совместимости с `scipy 1.12+`.
  - **Варианты улучшения на основе `librosa` (исследовать и выбрать оптимальный набор):**
    1. **Harmonic/Percussive Separation** (`librosa.effects.hpss`):
       - Разделить сигнал на гармоническую (мелодия, аккорды) и перкуссивную (ударные) составляющие.
       - Припев, как правило, имеет более высокую энергию обеих составляющих.
       - Использовать как дополнительный признак для подтверждения сегментов-кандидатов.

       - Припевы часто имеют схожие хроматические паттерны при каждом повторении.
       - Self-similarity по хроме позволяет найти повторяющиеся структуры.
    3. **Self-Similarity Matrix** (`librosa.segment.recurrence_matrix`):
       - Матрица самоподобия показывает, насколько похожи друг на друга временны́е фреймы.
       - Блоки высокого сходства вдоль диагонали соответствуют повторяющимся секциям (припевам).
       - Можно использовать совместно с chroma или MFCC для построения матрицы.
    4. **Tempogram / Ритмическая стабильность** (`librosa.feature.tempogram`):
       - Темпограмма отражает ритмическую активность во времени.
       - Припевы, как правило, имеют более стабильный и выраженный ритм по сравнению с куплетами и бриджами.
