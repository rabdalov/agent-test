# ADR-002: Отказ от GPU-ускорения (CPU-only подход)

**Статус:** ✅ Принято  
**Дата:** 2026-03-19  
**Итерация:** 49  

---

## Контекст

Проект караоке-бота имеет доступ к AMD GPU (Rembrandt Radeon 680M). Возник вопрос о целесообразности использования GPU-ускорения для ML-задач (в первую очередь `demucs` для разделения аудио).

## Решение

**Не использовать GPU нигде в проекте. Использовать максимально быстрый и лёгкий стек на CPU-only.**

## Обоснование

### 1. Масштаб проекта

- **1-2 пользователя максимум** — низкая нагрузка
- Последовательная обработка треков — нет конкуренции за ресурсы
- Общее время пайплайна: 3-5 минут, при этом GPU даёт выигрыш только на одном шаге

### 2. Производительность: CPU vs GPU

| Задача | CPU | GPU | Выигрыш GPU | Критично? |
|--------|-----|-----|-------------|-----------|
| `demucs` (3-4 мин трек) | 30-60 сек | 10-20 сек | ~30-40 сек | ❌ Нет |
| `librosa` | Полная скорость | Нет поддержки | — | — |
| `msaf` | Приемлемо | Частично | Незначителен | ❌ Нет |
| Транскрибация | N/A (внешний API) | N/A | — | — |
| Рендеринг видео | Приемлемо | NVENC/VA-API | Незначителен | ❌ Нет |

**Вывод:** Экономия 30-40 секунд на фоне общего времени пайплайна 3-5 минут **некритична**.

### 3. Стабильность и сложность

Согласно [ADR-001](ADR-001.%20UV_vs_Conda.md), приоритет проекта — **максимальная стабильность инфраструктуры**.

| Фактор | CPU-only | ROCm (AMD) | CUDA (NVIDIA) |
|--------|----------|------------|---------------|
| **Сложность инфраструктуры** | ⭐ Простая | ⭐⭐⭐ Сложная | ⭐⭐ Средняя |
| **Стабильность** | ⭐⭐⭐ Максимальная | ⭐⭐ Средняя | ⭐⭐⭐ Высокая |
| **Размер Docker образа** | ~550 MB | ~5 GB | ~1.5 GB |
| **Время сборки** | Быстро | Медленно | Средне |
| **Зависимости хоста** | Нет | ROCm runtime | NVIDIA drivers |
| **Отладка проблем** | Просто | Сложно | Средне |
| **Точки отказа** | Минимум | Много | Средне |

### 4. Специфика AMD GPU

- **ROCm для iGPU** (Rembrandt 680M): частичная поддержка, не гарантирует прироста
- **Образ с ROCm**: ~5 GB (в 10 раз больше CPU-only)
- **Настройка**: требуется ROCm runtime на хосте, доступ к `/dev/kfd` и `/dev/dri`

Для AMD iGPU **CPU-only стабильнее и проще**.

### 5. Соответствие ADR-001

> Приоритет проекта — **максимальная стабильность инфраструктуры**. Любое усложнение добавляется только при появлении конкретной проблемы.

GPU-ускорение:
- Добавляет сложности (runtime, драйверы, объём образа)
- Не решает критичной проблемы (30 сек экономии — не критично)
- Уменьшает стабильность (больше точек отказа)

## Последствия

### Положительные

- ✅ **Максимальная стабильность** — минимум точек отказа
- ✅ **Компактные образы** — ~550 MB вместо ~5 GB
- ✅ **Быстрая сборка** — нет тяжёлых ML-фреймворков с GPU-поддержкой
- ✅ **Простота деплоя** — работает на любой машине с Docker
- ✅ **Воспроизводимость** — детерминированное поведение без привязки к железу
- ✅ **Соответствие принципу KISS** — простота важнее сложности

### Отрицательные

- ⚠️ demucs работает медленнее на ~30-40 секунд
- ⚠️ При масштабировании на 10+ пользователей потребуется пересмотр

## Реализация

### Базовый образ

**`ubuntu:24.04`** — выбран как базовый образ для всех Docker-контейнеров проекта.

Обоснование выбора:
- LTS поддержка до 2029 года
- Лучшая совместимость с ROCm (если когда-нибудь понадобится экспериментировать)
- Стабильнее Debian 13 (Testing)
- Идентичные версии пакетов (Python 3.12, FFmpeg 6.1)

### Dockerfile (CPU-only)

```dockerfile
FROM ubuntu:24.04

# PyTorch CPU-only — стабильно и компактно
RUN uv pip install torch torchvision torchaudio \
    --index-url https://download.pytorch.org/whl/cpu

# demucs будет использовать CPU
RUN uv add demucs
```

### Конфигурация

| Параметр | Значение | Пояснение |
|----------|----------|-----------|
| **Базовый образ** | `ubuntu:24.04` | LTS, стабильность |
| **PyTorch** | CPU-версия | `--index-url https://download.pytorch.org/whl/cpu` |
| **GPU** | Не используется | Явно отключено |
| **Размер образа** | ~550 MB | Компактно |

## Исключения

В проекте **нет исключений** — GPU не используется нигде:
- ❌ `demucs` — CPU-only
- ❌ `librosa` — CPU-only (GPU не поддерживается)
- ❌ `msaf` — CPU-only
- ❌ Транскрибация — внешний API (GPU на стороне провайдера)
- ❌ Рендеринг видео — CPU (без NVENC/VA-API)

## Будущее

Если в проекте появятся задачи, критичные к GPU:

1. **Оценить реальный прирост** — замерить экономию времени vs усложнение
2. **Для 1-2 пользователей** — скорее всего, CPU останется оптимальным
3. **При масштабировании** — рассмотреть NVIDIA (CUDA стабильнее ROCm)
4. **Сохранить CPU fallback** — всегда иметь работающий CPU-only вариант
