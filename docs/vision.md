Техническое видение проекта
============================

## Технологии, управление зависимостями и сборкой

- **Язык и версия Python**  
  - Проект реализован на **Python 3.12**.  
  - Код пишется с использованием типизации и простого ООП‑подхода (1 класс = 1 файл).

- **Управление зависимостями: `uv` + `pyproject.toml`**  
  - В корне проекта находится `pyproject.toml`, описывающий все зависимости и конфигурацию проекта.  
  - Все Python‑зависимости (включая `demucs`) управляются через `uv`:
    - `uv add <package>` — добавление зависимостей,
    - `uv sync` — установка/обновление зависимостей,
    - `uv run ...` — запуск команд в окружении проекта.  
  - Отдельный `pip` напрямую не используется.

- **Основные Python‑зависимости (runtime)**  
  - `aiogram` — работа с Telegram Bot API (режим polling).  
  - `openai` — клиент для работы с LLM через провайдер OpenRouter.  
  - `demucs` — разделение голоса и музыки, устанавливается как Python‑пакет через `uv` и используется как CLI или модуль.  
  - Дополнительно (минимально необходимое):
    - `pydantic` — для описания и валидации конфигурации/моделей данных.
    - `httpx` — для обращения к внешнему сервису транскрибации (`speeches.ai`), если недостаточно встроенных возможностей `openai` клиента.

- **Системные и внешние зависимости**  
  - `ffmpeg` — обязательная системная зависимость для работы с аудио/видео.  
  - Доступ к внешнему Whisper‑совместимому сервису транскрибации (`speeches.ai`).  
  - Локально установленный `demucs` (как Python‑пакет в окружении проекта через `uv`).  

- **Сборка и запуск локально (без Docker)**  
  - Основные команды запуска выполняются напрямую через `uv`:
    - запуск бота: `uv run python -m <имя_модуля_бота>`,  
    - вспомогательные скрипты и утилиты также запускаются через `uv run`.  
  - Для Unix/WSL **опционально** используется простой `Makefile`:
    - `make run` — запуск бота,
    - дополнительные цели (`make dev`, `make lint` и т.п.) добавляются только при реальной необходимости.  
  - Под Windows основным способом остаются прямые команды `uv` (без обязательного использования `make`).

- **Сборка и запуск в Docker**  
  - Один простой `Dockerfile` на базе официального образа Python 3.12 (вариант `-slim`).  
  - В контейнере устанавливаются:
    - `uv`,
    - зависимости проекта (`uv sync`),
    - системные зависимости (`ffmpeg`, при необходимости дополнительные).  
  - Запуск бота в контейнере — через `uv run ...`.  
  - Для удобства могут быть цели `make docker-build` и `make docker-run` (для Unix/WSL), но они не являются обязательными для Windows.

## Принципы разработки

- **Простота (KISS)**  
  - Приоритет — максимально простые решения.  
  - Любое усложнение (новый слой абстракции, паттерн, сервис) добавляется только при появлении конкретной проблемы (дублирование, сложность поддержки, тестирования и т.п.).

- **ООП и 1 класс = 1 файл**  
  - Каждый значимый объект домена (бот, пайплайн обработки, работа с аудио, интеграции с внешними сервисами) оформлен как отдельный класс в отдельном файле.  
  - Наследование используется минимально, предпочтение отдаётся композиции простых объектов.

- **Три простых слоя без сложных архитектур**  
  - Слой Telegram‑интерфейса (обработка апдейтов и команд).  
  - Слой доменного пайплайна (последовательность шагов обработки запроса пользователя).  
  - Слой интеграций (обёртки над внешними и локальными сервисами: `speeches.ai`, `demucs`, LLM, файловая система и т.п.).  
  - Не используем сложные архитектурные подходы (чистая архитектура, CQRS и т.п.), пока в них нет явной необходимости.

- **Асинхронность как базовый подход**  
  - Основной стиль разработки — асинхронный (`async/await`), что соответствует работе `aiogram` и сетевым вызовам к внешним сервисам.  
  - Блокирующие операции (например, тяжёлые CPU‑задачи или синхронные библиотеки) по возможности выносятся в отдельные процессы/пулы, чтобы не блокировать обработку Telegram‑апдейтов.

- **Конфигурация через окружение и файлы**  
  - Все переменные среды выполнения (токены, ключи, URL сервисов, режимы работы) задаются через переменные окружения и/или отдельный конфигурационный файл.  
  - Жёстко зашитые в код значения (кроме по-настоящему константных вещей) избегаются.

- **Прозрачная структура кода и обработка ошибок**  
  - Структура каталогов и модулей должна быть очевидной и предсказуемой, без лишних уровней вложенности.  
  - Минимизируются фоллбэки и избыточные проверки: если данные не соответствуют ожиданиям, обработка прерывается.  
  - При любом существенном несоответствии данных пайплайн прерывается, а пользователю возвращается понятное и максимально подробное сообщение об ошибке, которое помогает скорректировать ввод или повторить попытку.

## Структура проекта и архитектура

- **Структура проекта**  
  - В корне репозитория находятся:
    - `pyproject.toml` — описание проекта и зависимостей (через `uv`),
    - `README.md`, каталог `docs/` (включая `docs/vision.md`, `docs/idea.md`),
    - `Dockerfile` — образ для локального запуска,
    - `Makefile` — опциональные команды для Unix/WSL (build/run).  
  - Основной код бота размещается в пакете `app/` с одноуровневой структурой:
    - `app/__init__.py`,
    - `app/main.py` — точка входа, инициализация настроек, зависимостей и запуск бота,
    - `app/config.py` — класс `Settings` для загрузки конфигурации из окружения/файла,
    - `app/models.py` — доменные модели (запрос пользователя, состояние пайплайна, результат обработки и т.п.),
    - `app/bot_app.py` — класс `BotApp`, инициализирующий `aiogram`‑бота и регистрирующий хендлеры,
    - `app/handlers_karaoke.py` — класс `KaraokeHandlers` с обработчиками команд и сообщений,
    - `app/pipeline.py` — класс `KaraokePipeline`, реализующий последовательность шагов обработки,
    - `app/speeches_client.py` — класс `SpeechesClient` для работы с `speeches.ai`,
    - `app/demucs_service.py` — класс `DemucsService` для разделения голоса и музыки,
    - `app/llm_client.py` — класс `LLMClient` для работы с LLM через `openai`/OpenRouter.
    - `app/correct_transcript_service.py` — класс `CorrectTranscriptService` для корректировки транскрипции с использованием LLM.  
  - В каждом файле содержится один основной класс, вспомогательные функции относятся только к этому классу.

- **Высокоуровневая архитектура**  
  - Слой Telegram‑интерфейса (`BotApp`, `KaraokeHandlers`) отвечает за приём сообщений от пользователя и отправку результатов.  
  - Доменный пайплайн (`KaraokePipeline`) реализует основной сценарий обработки запроса пользователя.  
  - Классы интеграций (`SpeechesClient`, `DemucsService`, `LLMClient`) инкапсулируют работу с внешними и локальными сервисами.

- **Пайплайн обработки и шаги**  
  - `KaraokePipeline` реализует последовательность шагов:
    1. приём и валидация входных данных (файл/ссылка),
    2. загрузка и подготовка аудио/видео,
    3. получение текста песни,
    4. разделение голоса и музыки через `DemucsService`,
    5. транскрибация голосовой дорожки через `SpeechesClient`,
    6. корректировка транскрипции с использованием LLM (опционально, через `CorrectTranscriptService`),
    7. сопоставление текста с таймкодами слов,
    8. генерация `.ass` и финального караоке‑видео с использованием `ffmpeg`.
   
   На шаге `RENDER_VIDEO` создаётся один MP4-файл, содержащий **три переключаемые аудиодорожки**:
   
   1. **Instrumental** — чистая инструментальная дорожка (без вокала)
   2. **Original** — полный исходный трек с вокалом (берётся из `track_source`)
   3. **Instrumental + Voice (40%)** — микс: инструментальная дорожка + вокал на 40% громкости (настраивается через `AUDIO_MIX_VOICE_VOLUME` в .env)
   
   Пользователь может переключаться между дорожками в любом видеоплеере (как переключение языка озвучки или субтитров).
  - Каждый шаг пайплайна является **целостным и законченным**: получает на вход необходимые данные (включая результаты предыдущих шагов) и возвращает результат шага в виде одного или нескольких файлов.  
  - Все промежуточные файлы для конкретного трека сохраняются в отдельной локальной папке, которая при запуске в Docker монтируется в контейнер через `volume`, чтобы результаты шагов были доступны между перезапусками и для продолжения обработки.
  - Пайплайн поддерживает:
    - стандартное последовательное выполнение шагов,
    - продолжение с последнего неуспешного шага (при наличии сохранённых промежуточных файлов),
    - ручной запуск обработки с конкретного шага (при условии наличия артефактов предыдущих шагов).

## Модель данных и сценарии работы

- **Основные сущности и состояния**  
  - Для каждого запроса пользователя генерируется `track_id` (строка/UUID), по которому:
    - создаётся отдельная папка с промежуточными файлами трека,
    - сохраняется состояние пайплайна и итоговый результат.
  - Перечень ключевых доменных моделей:
    - `UserRequest` — запрос пользователя:
      - `user_id` (Telegram user id),
      - `track_id`,
      - `source_type` (`file` или `yandex_link`),
      - `source_url_or_file_path` (путь к локальному файлу или исходная ссылка),
      - при необходимости дополнительные поля (язык, название и т.п.).
    - `PipelineStep` (Enum) — шаги пайплайна:
      - `DOWNLOAD`,
      - `GET_LYRICS`,
      - `SEPARATE`,
      - `TRANSCRIBE`,
      - `CORRECT_TRANSCRIPT`,
      - `ALIGN`,
      - `GENERATE_ASS`,
      - `RENDER_VIDEO`.
    - `PipelineState` — состояние выполнения пайплайна (Pydantic‑модель, персистируется в `state.json` после каждого изменения):
      - `track_id` — идентификатор трека (заполняется при создании),
      - `current_step` — текущий или последний успешно выполненный шаг (`PipelineStep | None`, заполняется при старте каждого шага),
      - `status` — статус выполнения: `pending`, `in_progress`, `failed`, `completed` (обновляется на каждом переходе),
      - `error_message` — сообщение об ошибке при `status = failed` (заполняется при исключении),
      - `track_file_name` — оригинальное имя файла трека (заполняется на шаге `DOWNLOAD`),
      - `track_source` — источник трека: URL или путь к локальному файлу (заполняется на шаге `DOWNLOAD`),
      - `track_stem` — базовое имя трека без расширения, используется как префикс для артефактов (заполняется на шаге `DOWNLOAD`),
      - `vocal_file` — путь к файлу вокальной дорожки (заполняется на шаге `SEPARATE`),
      - `instrumental_file` — путь к файлу инструментальной дорожки (заполняется на шаге `SEPARATE`),
      - `transcribe_json_file` — путь к JSON с результатом транскрибации (заполняется на шаге `TRANSCRIBE`),
      - `corrected_transcribe_json_file` — путь к скорректированному JSON с результатом транскрибации (заполняется на шаге `CORRECT_TRANSCRIPT`, используется шагом `ALIGN`),
      - `source_lyrics_file` — путь к TXT с текстом песни (заполняется на шаге `GET_LYRICS`),
      - `aligned_lyrics_file` — путь к JSON с выровненным текстом и таймкодами (заполняется на шаге `ALIGN`),
      - `ass_file` — путь к ASS‑файлу субтитров (заполняется на шаге `GENERATE_ASS`),
      - `output_file` — путь к итоговому MP4‑файлу (заполняется на шаге `RENDER_VIDEO`).
    - `PipelineResult` — итог выполнения:
      - `track_id`,
      - `status`,
      - `final_video_path` (при успехе),
      - `error_message` (при неудаче).

- **Форматы файлов и хранение состояния**  
  - Для каждого трека создаётся своя папка, внутри которой хранятся:
    - исходный медиафайл: **MP3/FLAC/AVI/MP4/MKV**,
    - раздельные дорожки голоса и музыки: **MP3 320 kbps / 44100 Hz**,
    - результат транскрибации: **JSON**,
    - текст песни: **TXT**,
    - результат выравнивания текста по таймкодам: **JSON**,
    - файл субтитров: **ASS**,
    - финальное караоке‑видео: **MP4**,
    - файл состояния пайплайна и прогресса по треку: **JSON** (без использования базы данных).  
  - Работа с треком и его шагами всегда опирается на содержимое этой папки и JSON‑состояние.

- **Сценарии работы пользователя**  
  - **Базовый сценарий (полный пайплайн)**  
    - Пользователь отправляет боту файл с композицией или ссылку на трек в Яндекс‑Музыке.  
    - Бот:
      - создаёт `track_id` и папку трека,
      - сохраняет/скачивает исходный медиафайл,
      - формирует `UserRequest` и запускает `KaraokePipeline`.  
    - Пайплайн последовательно проходит шаги `DOWNLOAD → GET_LYRICS → SEPARATE → TRANSCRIBE → CORRECT_TRANSCRIPT → ALIGN → GENERATE_ASS → RENDER_VIDEO`, на каждом шаге:
      - читает нужные входные файлы из папки трека,
      - записывает результаты в новые файлы,
      - обновляет `PipelineState` в JSON,
      - отправляет пользователю краткое сообщение о прогрессе (успех/ошибка шага).  
    - По завершении пользователю отправляется итоговое MP4‑видео или подробное сообщение об ошибке.

  - **Получение текста песни (шаг GET_LYRICS)**  
    - Сначала бот пытается автоматически найти текст с помощью внешней библиотеки/сервиса (например, `Lyrica` или `lyrics-lib`), используя доступные метаданные трека.  
    - Если текст найти не удалось или качество результата неудовлетворительное, бот просит пользователя прислать полный текст песни в виде простого текста.  
    - Результирующий текст песни сохраняется в TXT‑файл в папке трека и используется на следующих шагах.

  - **Продолжение после ошибки**  
    - Если на одном из шагов пайплайна произошла ошибка, в JSON‑состоянии сохраняются:
      - последний успешно пройденный шаг,
      - шаг, на котором произошла ошибка,
      - сообщение об ошибке.  
    - Пользователь может вызвать команду повторного запуска (например, `/continue`), после чего:
      - `KaraokePipeline` считывает состояние последнего неуспешного запроса этого пользователя,
      - при наличии всех необходимых промежуточных файлов продолжает выполнение с проблемного шага.  
    - На первом этапе поддерживается продолжение только **последнего неуспешного запроса** пользователя, без явного выбора `track_id`.

## Работа с внешними и локальными сервисами, LLM и конфигурацией

- **Внешний сервис транскрибации (`speeches.ai`)**  
  - Для транскрибации голосовой дорожки используется внешний Whisper‑совместимый сервис `speeches.ai`.  
  - Класс `SpeechesClient` инкапсулирует:
    - формирование HTTP‑запросов (через `httpx` или аналогичный async‑клиент),
    - передачу аудиофайла или ссылки,
    - обработку ответов и сохранение результата транскрибации в JSON‑файл.  
  - Адрес сервиса и ключи доступа задаются через конфигурацию (переменные окружения/конфиг‑файл).

- **Локальный сервис разделения голоса и музыки (`demucs`)**  
  - `DemucsService` отвечает за вызов `demucs` как локального инструмента:
    - запуск через CLI или модуль Python в рамках того же виртуального окружения (`uv`),
    - сохранение раздельных дорожек (voice/music) в формате MP3 320 kbps / 44100 Hz в папке трека.  
  - Путь к `demucs` и параметры запуска (модель, уровни шума и т.п.) задаются через конфигурацию.

- **Интеграция с LLM через OpenRouter (`openai` клиент)**  
  - Класс `LLMClient` использует официальный Python‑клиент `openai`, настроенный на работу через провайдера OpenRouter.  
  - Основные сценарии использования LLM:
    - корректировка транскрипции на основе текста песни (шаг `CORRECT_TRANSCRIPT`),
    - помощь в уточнении и нормализации текста песни (если это необходимо),
    - при желании — помощь в выравнивании текста по таймкодам или генерации подсказок по ошибкам.  
  - Ключи и URL OpenRouter передаются через конфигурацию, конкретные модели и промпты задаются минимально необходимым образом.

- **Интеграция с Telegram Bot API (`aiogram`, polling)**  
  - Бот реализован на базе `aiogram` с использованием режима polling.  
  - `BotApp` отвечает за:
    - инициализацию бота по токену из конфигурации,
    - регистрацию хендлеров,
    - запуск цикла polling.  
  - Вебхуки и дополнительные механизмы доставки обновлений не используются для упрощения.

- **Конфигурация**  
  - Все параметры окружения (токены, ключи API, URL сервисов, пути к утилитам, режимы работы) описываются в одном классе `Settings` (Pydantic) в `app/config.py`.  
  - Значения берутся из:
    - переменных окружения,
    - при необходимости — из одного конфигурационного файла (например, `.env` или YAML/JSON, загружаемого при старте).  
  - В коде отсутствуют жёстко зашитые секреты и URL: всё, что может меняться между окружениями, задаётся через конфигурацию.

## Логирование, сборка и запуск

- **Логирование**  
  - Используется стандартный модуль `logging` без дополнительных обёрток и фреймворков.  
  - Настройка логгера выполняется один раз при старте приложения (в `main.py` или `bot_app.py`):
    - формат логов включает время, уровень и краткое сообщение,
    - по умолчанию уровень `INFO`, с возможностью переключения на `DEBUG` через конфигурацию.  
  - Логируются:
    - ключевые события пайплайна (старт/завершение шагов, ошибки),
    - обращения к внешним сервисам (успех/ошибка, время выполнения),
    - системные ошибки и исключения.  
  - Логи выводятся в стандартный вывод, что упрощает просмотр как локально, так и в Docker.

- **Сборка и запуск локально (без Docker)**  
  - Управление зависимостями через `uv`:
    - установка/обновление: `uv sync`,
    - запуск бота: `uv run python -m app.main`.  
  - Для Unix/WSL предусмотрен простой `Makefile`:
    - `make run` — запуск бота,
    - при необходимости могут быть добавлены минимальные дополнительные цели (`make lint`, `make fmt`), но только по мере реальной надобности.  
  - Под Windows основным способом остаётся прямой запуск через `uv`, без обязательного использования `make`.

- **Сборка и запуск в Docker**  
  - Используется один простой `Dockerfile`:
    - базовый образ `python:3.12-slim`,
    - установка `uv` и зависимостей проекта (`uv sync`),
    - установка системных утилит (`ffmpeg`, при необходимости других минимально необходимых инструментов),
    - настройка непривилегированного пользователя и рабочей директории.  
  - Запуск приложения в контейнере осуществляется командой `uv run python -m app.main`.  
  - Для разработки предполагается локальный Docker‑запуск, без обязательного удалённого деплоя:
    - монтирование папки с треками как volume,
    - монтирование папки с заранее загруженными моделями `demucs` как отдельного volume **или** фоновая загрузка моделей после первого запуска контейнера (а не на этапе сборки образа), чтобы не раздувать базовый образ,
    - передача конфигурации через переменные окружения.  
  - Для Unix/WSL при желании могут быть добавлены цели:
    - `make docker-build` — сборка образа,
    - `make docker-run` — запуск контейнера с нужными volume и env‑переменными.
